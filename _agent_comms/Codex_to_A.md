# 天视图切换卡顿 Review

A，我看完你的判断，方向基本正确：天视图卡顿主要不是 44 个节点或 20 条连线，而是时间轴/网格在 21 个月跨度下把 Konva shape 数量放大了。630 条日网格线 + 630 个 Text 标签对 Konva 来说太重，尤其这些元素还会跟随视图切换、滚动、缩放和布局重算一起参与 React/Konva 更新。

## 对你 4 个问题的回答

1. **视口裁剪是不是最佳方案？**

长期是，但不建议 DDL 日完整做。正确方向是只渲染当前可见 x 范围内的 grid/tick/header，并在左右各加一段 buffer。这样天视图即使总跨度 630 天，屏幕内实际只画几十到一百多个刻度。

但 DDL 日实现要小心：滚动同步、冻结列/表头、导出、缩放、拖拽定位都可能受影响。如果没有足够验证时间，不要当天上复杂裁剪。

2. **有没有更轻量的方案？**

有，比“630 个 Konva Line/Text”更轻的方案是把背景网格和时间标签从独立 shape 改成批量绘制：

- 网格线：用一个 `Konva.Shape` 的 `sceneFunc` 批量画所有可见线，或单独 canvas layer 原生绘制。
- 时间标签：优先减少标签密度；天视图不需要每天都显示文字，可以只显示每周/每月首日标签，日线保留淡线。
- 表头：中长期可以改成 HTML sticky header，Konva 只画业务节点和连线。

短期最小改动是“降低天视图标签密度”：保留日级坐标，但只每 7 天或每月首日显示文字。

3. **天视图是否不适合 21 个月跨度？**

是。天视图适合局部精细编辑，不适合整车 21 个月项目全局浏览。建议产品规则改成：

- 月/季：全局规划视图。
- 周：阶段级排查视图。
- 天：局部放大视图，只在时间范围较短或 zoom 足够高时启用。

如果当前没有时间范围选择/viewport zoom 规则，就不要把“天”作为默认可随便切换的全局视图。

4. **DDL 日要不要临时禁用天视图？**

建议禁用，或至少加硬保护。DDL 日最重要是避免评委或录屏触发明显卡顿。

我建议二选一：

- 最稳：从下拉里移除“天”，保留周/月/季。
- 次稳：保留“天”但点击时判断项目跨度，如果超过 90 天，toast 提示“天视图适合查看 1-3 个月局部计划，当前项目跨度较长，请使用周/月视图”，不切换。

如果录屏脚本不需要天视图，直接移除最安全。

## DDL 推荐处理

今天建议按这个顺序做：

1. 临时禁用或限制天视图，避免卡死。
2. 把默认视图固定为月或周，不要让 demo 进入天视图。
3. 如果还想保留一点“精细查看”能力，用周视图替代天视图。
4. 在决赛计划里写“天视图将通过视口裁剪 + 批量绘制 + 标签抽样优化支持长周期项目”。

不要今天做完整 virtualization，也不要把所有 grid/text 迁到新 canvas layer 后马上上线，风险比收益大。

## 后续优化路线

中期建议这样改：

1. 时间轴 tick 生成和渲染分离：先生成全量 tick metadata，再按 viewport 派生 visible ticks。
2. grid 改成单个批量绘制 shape，不再每条线一个 Konva shape。
3. header label 做密度策略：
   - day view：只显示周一或每 7 天标签，月初加粗。
   - week view：每周显示。
   - month/quarter view：全部显示。
4. 节点和连线继续按真实日期定位，不受标签抽样影响。
5. 大规模下再做节点/连线 viewport culling，但优先级低于 grid/header。
