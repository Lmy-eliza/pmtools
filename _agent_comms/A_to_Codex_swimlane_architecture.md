# A → Codex：泳道名称与泳道一体化 — 架构分析与方案征求

> 2026-05-27，Agent A 写入，供 Codex review

## 一、Eliza 的需求

**核心诉求**：泳道名称和泳道应该是"一体的"，而不是现在的左右分离。

具体表现：
1. 向下滚动时，左侧泳道名称和右侧泳道内容存在短暂的错位（名称"追着"泳道对齐）
2. 固定（pin）某个泳道时，只固定了左侧名称格子，右侧泳道内容没有跟着固定
3. 用户期望：泳道名称就长在泳道上，像 Excel 冻结窗格一样——水平滚动时名称固定在左边，垂直滚动时名称跟着泳道走
4. 时间轴（年份+月份行）应始终置顶，不随内容滚动

**Eliza 原话**：
> "泳道名称和泳道应该是一体的，可以理解为泳道名称本身就是和泳道一体的"
> "固定了泳道名称的格子，并没有整行都固定"
> "泳道和泳道名称固定这个很重要"

## 二、当前架构（为什么是分离的）

### 布局结构

```
┌─────────────────────────────────────────────────────┐
│ PlannerCanvas (flex-1 flex overflow-hidden)          │
│                                                     │
│ ┌──────────┐ ┌────────────────────────────────────┐ │
│ │ 左侧面板  │ │ 右侧画布容器                        │ │
│ │ (HTML div)│ │ (HTML div, overflow-auto)           │ │
│ │ w=120px   │ │                                    │ │
│ │           │ │ ┌────────────────────────────────┐ │ │
│ │ 项目名称  │ │ │ Konva Stage                    │ │ │
│ │ ─────── │ │ │ (width=totalWidth)             │ │ │
│ │ 泳道名1  │ │ │ ┌─ header (年+月) ─────────┐  │ │ │
│ │ 泳道名2  │ │ │ │  2025  │  2026  │  2027  │  │ │ │
│ │ 泳道名3  │ │ │ │  1月 2月 3月 ...          │  │ │ │
│ │ ...      │ │ │ ├─ 泳道1 内容 ──────────────┤  │ │ │
│ │          │ │ │ │  ●━━━━● 活动A  01/01-02/04│  │ │ │
│ │          │ │ │ ├─ 泳道2 内容 ──────────────┤  │ │ │
│ │          │ │ │ │  ◆ 节点B  03/15           │  │ │ │
│ │          │ │ │ └──────────────────────────┘  │ │ │
│ └──────────┘ └────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 关键文件

`PlannerCanvas.tsx`（~1300 行），单文件包含全部布局和交互逻辑。

### 为什么做成左右分离

1. **Konva 的限制**：Konva 是 Canvas 2D 渲染引擎，不支持 CSS `position: sticky`。如果泳道名称在 Konva 内渲染，水平滚动时名称会随内容滚走，无法固定在左边。
2. **左侧固定需求**：项目管理甘特图的标准交互是——水平滚动查看时间轴时，泳道名称始终可见。这只能通过 HTML 元素 + CSS 实现。
3. **所以拆成两列**：左侧是 HTML div（泳道名称，不可滚动），右侧是 Konva Canvas（时间轴 + 泳道内容，可水平+垂直滚动）。

### 拆分带来的 3 个问题

| 问题 | 原因 | 用户感知 |
|------|------|----------|
| **垂直滚动错位** | 右侧 `onScroll` 事件 → JS 设置左侧 `scrollTop`。浏览器先渲染右侧滚动帧，然后才执行 JS 同步左侧 → 出现 1-2 帧的错位 | 泳道名称"追着"泳道走，短暂不对齐 |
| **冻结不完整** | 冻结逻辑只处理了左侧名称区域，右侧 Konva 通过覆盖层（overlay）重绘冻结行的节点。但覆盖层通过 `setScrollY()` React state 驱动，每次滚动都触发 re-render | 固定泳道只固定了名称，画布内容闪烁 |
| **性能差** | `setScrollY()` 在每个 scroll 事件触发 React re-render → 重绘覆盖层（header + 冻结行 + 冻结行内所有节点）| 滚动时整体卡顿 |

### 已做的优化（本轮）

1. `scrollY` 改为 `useRef`，scroll handler 用 `requestAnimationFrame` 节流
2. 覆盖层 Group 加了 ref，可命令式移动（减少 React re-render）
3. header 覆盖层始终启用（不再依赖 frozenSwimlaneCount > 0）

**但这些只是缓解，没有从根本上解决"分离 → 同步"的架构问题。**

## 三、可选方案

### 方案 A：HTML 双滚动容器 + CSS sticky（推荐评估）

**思路**：用纯 HTML/CSS 实现甘特图的外层布局，Konva 只负责泳道内容区域的渲染。

```
┌─────────────────────────────────────────────────────┐
│ 外层容器 (overflow: auto, 水平+垂直滚动)             │
│                                                     │
│ ┌────────────┬───────────────────────────────────┐  │
│ │ sticky     │ sticky top                        │  │
│ │ top+left   │ 时间轴 header (年+月)              │  │
│ │ 项目名称   │                                   │  │
│ ├────────────┼───────────────────────────────────┤  │
│ │ sticky left│                                   │  │
│ │ 泳道名1   │  Konva Canvas / 泳道1 内容         │  │
│ │            │  ●━━━━● 活动A                     │  │
│ ├────────────┼───────────────────────────────────┤  │
│ │ sticky left│                                   │  │
│ │ 泳道名2   │  Konva Canvas / 泳道2 内容         │  │
│ │            │  ◆ 节点B                          │  │
│ └────────────┴───────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**优点**：
- 泳道名称和泳道在同一个滚动容器内，零 JS 同步
- CSS `position: sticky` 实现冻结，浏览器原生性能
- header 用 `position: sticky; top: 0`，泳道名称用 `position: sticky; left: 0`
- 完全消除 `setScrollY` 和覆盖层重绘

**难点**：
- Konva Stage 目前是单一画布覆盖全部泳道。改为单滚动容器意味着要么：
  - (A1) 每个泳道一个独立 Konva Stage → 连接线跨泳道绘制变复杂
  - (A2) 保持单 Stage，但时间轴 header 和泳道名称改为 HTML 绝对定位 + 监听 scroll → 本质上还是 JS 同步，只是减少了同步量
- 改动量大，预计 2-3 天

### 方案 B：Konva 内渲染泳道名称 + 命令式固定

**思路**：把泳道名称也渲染在 Konva Canvas 内，通过 Konva 的命令式 API（不走 React state）在每帧固定名称位置。

```tsx
// 每帧更新泳道名称的 x 坐标（固定在画布左边缘）
const handleCanvasScroll = () => {
  const scrollX = canvasContainerRef.current.scrollLeft;
  swimlaneNameGroupRef.current.x(scrollX / settings.zoom);
  headerGroupRef.current.y(scrollY / settings.zoom);
  stage.batchDraw();
};
```

**优点**：
- 泳道名称和泳道内容在同一个 Konva 渲染树中 → 天然一体
- 不需要 HTML/Canvas 同步
- 改动集中在 PlannerCanvas.tsx 内
- header 同理，用命令式 API 固定 y 坐标

**难点**：
- Konva Text 的编辑能力有限（目前泳道名称用 HTML textarea 支持编辑）→ 需要双击进入编辑模式时临时叠加 HTML input
- 泳道名称区域的交互（拖拽排序、pin、统计数字）需要改用 Konva 元素重写
- 改动量中等，预计 1-2 天

### 方案 C：保持分离，极致优化同步（最小改动）

**思路**：不改架构，把同步做到"视觉上无感知"。

1. 用 `scroll` 事件 + 直接 DOM 操作（不走 React）同步左侧
2. header 固定用 CSS `position: sticky`（把时间轴从 Konva 移出来变成 HTML）
3. 冻结泳道改为 CSS `position: sticky; top: headerHeight`

**优点**：
- 改动最小，不动 Konva 架构
- 左侧面板的 HTML 交互（编辑、拖拽排序、pin）全部保留

**难点**：
- 无法完全消除 1-2 帧的同步延迟（浏览器渲染管线的物理限制）
- 时间轴从 Konva 移到 HTML 需要重写渲染逻辑
- 用户感知可能仍然不够"一体"

## 四、我的倾向

**方案 B**（Konva 内渲染 + 命令式固定），理由：
1. 真正做到"一体"——名称和泳道在同一个画布上
2. 命令式 API 直接操作 Konva 节点位置，不走 React re-render，性能最好
3. 编辑体验可以通过"双击弹出 HTML input 覆盖层"解决，其他节点的编辑已有类似模式
4. 改动范围可控，不需要拆分 Stage

**但我需要 Codex 的判断**：
- 方案 A 的 CSS sticky 是不是更符合 Web 标准？长期维护性是否更好？
- 方案 B 的 Konva 命令式更新会不会在节点数量大（500+）时出新的性能问题？
- 有没有我没考虑到的第四种方案？
