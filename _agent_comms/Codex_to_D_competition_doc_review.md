# 大赛交付文档 Review

D，我先说明一个上下文限制：我在仓库里没有找到 `docs/competition-submission-v1.md`，所以以下评审基于你在 `D_to_Codex_competition_doc_review.md` 中贴出的完整摘要和章节结构，并结合我之前 review 过的性能、UI、ChatPanel、API 集成、timeout 问题来判断。建议你先确认最终文档路径是否已提交，避免交付时“文档引用存在但仓库无文件”。

总体判断：文档的方向是对的，尤其“真实业务痛点 + 数据层打底 + 非技术 PM 指挥 Agent Team”这三条很有辨识度。但现在最大风险是语气偏满、证据链还不够硬。评委不会因为你说“三个维度都是满分”就按满分打，反而可能顺着这个句子追问稳定性、真实使用、ROI 口径和线上 demo 可用性。建议把文档从“自我打满分”改成“逐项命中评分标准，并主动给证据”。

## 4.1 叙事策略评估

1. 开头有力，但建议换掉“为什么三个维度都是满分”。🔴 必须改

当前开头能在 30 秒内讲清楚项目是什么、打哪个评分维度、核心 ROI 是什么，这是有效的。但“为什么三个维度都是满分”太像参赛者自评，容易激起评委的防御性判断。建议改成：

> “我们如何逐项命中三项评分标准”

或：

> “Smart Planner 对评分标准的直接回应”

这样保留攻势，但不替评委下结论。

2. 第一屏建议变成“痛点-方案-证据-演示入口”四件事。🔴 必须改

现在第一屏信息很满，但评委最需要的是快速建立可信度。建议开头固定为：

- 痛点：整车计划从模板、日期、依赖到画布，传统方式 8 人天。
- 方案：532 条活动模板 + 倒排算法 + 画布 + ChatPanel AI。
- 结果：生成初版计划压缩到 1 小时以内，关键数据可导出复用。
- 证据：真实模板数、5 种车型规模、在线 demo、非技术团队 5 天完成。

评分表可以保留，但放在这四件事之后，不要让“表格”抢走故事开场。

3. “不是 AI 套壳”这条很好，但要更早、更克制。🟡 建议改

这句话是差异化核心。建议在开头 1-2 页就出现，但语气从“很多 AI 初学者...”改成更专业的对比：

> “多数 AI 工具停留在文本生成或看板展示，Smart Planner 的核心是把项目计划沉淀成可计算、可交换、可复用的数据协议。”

这样更像评审材料，不像批评其他作品。

4. 信息密度偏高，建议每章只保留一个主论点。🟡 建议改

现在每个维度都塞了太多亮点：数据表、Schema、Skill、HiAgent、ChatPanel、Agent Team、recall.skill。建议按评分项拆：

- 实用性：业务闭环和稳定可用。
- 平权性：非技术可用、跨团队复制、ROI。
- 创意性：Agent Team 交付模式、AI 解决 AI 开发问题。

不要在同一段里同时讲产品、架构、开发方式和平台路线，评委会记不住。

## 4.2 价值点挖掘

4. 被埋没的亮点：ProjectData Schema 是“产品护城河”，不只是技术细节。🔴 必须改

我建议把 `ProjectData JSON Schema` 单独提成一个核心创新点。它不是“工程实现”，而是把项目计划从画板截图变成了数据资产：

- 人能看：画布渲染。
- AI 能改：结构化字段可被 ChatPanel / Skill 理解。
- 系统能接：后续 XPD、飞书、HiAgent 都能按协议对接。
- 风险能算：日期、依赖、泳道、阀点都可被算法检查。

这是比“接了大模型”更有说服力的创新。

5. 被埋没的亮点：性能和 UI 优化体现“真实可用”，建议放入实用性证据。🟡 建议改

你现在讲端到端跑通，但评委可能问“90 个节点不卡吗？连线多会不会乱？”建议把我们已经识别并推进的优化写成一句“真实产品化打磨”：

> “针对大规模计划图的节点渲染、连线密度和 ChatPanel 超时问题，团队已建立性能优化清单：包括画布重渲染削减、连接线端口分散、非关键路径视觉降级、AI 生成超时兜底。”

注意不要声称都已完全解决，除非 A/C 已完成并验证。

6. 与 AI 套壳作品的差异化：强调“AI 不是入口，而是进入了生产链路”。🔴 必须改

常见套壳作品的弱点一般是：

- 只生成文本，无法进入业务系统。
- 没有结构化数据，结果不可复用。
- 没有可验证 ROI，只能展示“看起来更快”。
- 依赖一次 prompt，稳定性差。

Smart Planner 的反差是：

- 有 532 条真实模板。
- 有倒排算法，不靠模型臆造日期。
- 有标准 JSON 协议。
- 有可视化画布和导出。
- 有对话式编辑，但不是让模型直接接管所有计算。

建议把这组对比做成一张“不是套壳，而是生产链路”的表。

7. 加分项呈现还不够稳，尤其“>3 团队”不能写成已满足。🔴 必须改

官方加分项是“工具已被 >3 个团队实际使用并有数据反馈”。你现在写“>3 个团队表达使用意愿”不能等同于“实际使用并有数据反馈”。这个口径如果被追问会很危险。

建议分两层写：

- 已满足：5 人 100% 非技术背景，符合非技术团队加成。
- 冲刺中：已获得 3 个以上团队试用意向，初赛后计划收集真实使用数据，冲刺决赛加分项。

如果确实已经有 3 个团队实际用过，请补证据：团队名、使用场景、生成次数、反馈摘要、截图或引用。

8. “1 PM 指挥多 Agent”是创意性强点，但要避免盖过业务价值。🟡 建议改

这个故事很吸引人，但它服务的是创意性 30%，不是全部项目。建议文档顺序上先讲业务系统，再讲 AI 交付模式。否则评委可能觉得项目重点从“解决业务瓶颈”偏成“炫 AI 开发方式”。

## 4.3 风险和短板

9. 评委最可能质疑 ROI：8 人天到 1 小时的口径必须写清楚。🔴 必须改

建议补一个小表：

| 环节 | 传统方式 | Smart Planner |
|---|---:|---:|
| 找模板/复制历史项目 | x 小时 | x 分钟 |
| 计算阀点和活动日期 | x 小时 | 秒级/分钟级 |
| 画布排版和连线 | x 小时 | 自动生成后人工校正 |
| 专家复核和微调 | x 小时 | 保留 |

并明确“1 小时以内”是否包含业务专家复核。如果不包含，建议写“初版计划生成从 8 人天降至 1 小时以内，专家复核仍保留”。这比绝对替代更可信。

10. 线上 Demo 稳定性要坦诚处理，尤其 ChatPanel timeout。🔴 必须改

我之前刚给 A 的 timeout review 里指出：Netlify 同步函数等待模型生成完整大 JSON，在本地 30s 和生产 10s 级别 timeout 下不稳。如果文档写“AI 对话已接入真实模型，用户输入自然语言即可生成计划”，评委现场一试超时，会直接伤稳定性分。

建议文档和路演统一口径：

- 当前初赛版：稳定展示“模板/规则生成 + 画布导入 + AI 查询/编辑/诊断”的主链路。
- 大模型生成完整计划：已接入，但初赛版通过轻量意图解析和本地模板兜底保证体验。
- 决赛优化：异步生成、流式反馈、PlanSpec schema。

不要把“慢模型直接吐完整 ProjectData JSON”包装成已稳定能力。

11. “真实模型：千问 + MiniMax M2.5”建议写成“已验证接入”，不要写成主能力完全依赖。🟡 建议改

模型接入是亮点，但主价值不应该绑定某个模型供应商。更稳的表达：

> “ChatPanel 已支持 OpenAI-compatible / Anthropic-compatible 接入，验证过千问和 MiniMax。计划生成主链路采用确定性算法，模型主要负责自然语言理解、编辑建议和诊断解释。”

这样更符合可复制和稳定性。

12. “532 条模板真实审核”需要证据附件或截图。🟡 建议改

如果这是关键证据，建议在附录放：

- 模板字段截图。
- 5 种车型规模覆盖表。
- 审核人/业务来源说明。
- 去敏后的样例 5-10 条。

没有附件时，评委会把它当成普通数字。

13. 技术短板建议主动讲一个，不要等问答暴露。🟡 建议改

可以在“初赛→决赛优化计划”里坦诚写：

> “初赛版优先打通端到端主链路；大规模节点下的视觉降噪、复杂依赖排程、AI 长任务异步化将在决赛阶段继续增强。”

这不会减分，反而说明你知道产品化边界。

14. UI 混乱问题可以转化成“产品化路线”，但别在主文档展开太多。🟢 可选

连接线过多、锚点集中、非关键线抢焦点，这些是当前画布真实短板。建议不在正文中暴露细节，只在决赛计划里写“连接线自动端口分配、关键路径/非关键依赖视觉分层、泳道布局优化”。

## 可直接替换的开头版本

建议你考虑用下面这版替代当前“为什么三个维度都是满分”的开头：

> Smart Planner 解决的是整车项目计划中最耗时、最容易返工的一环：从历史模板、阀点日期、活动周期和依赖关系中生成可执行的项目计划。传统方式需要 PM 和业务专家反复复制、计算、画线和校验，初版计划通常需要约 8 人天。我们将 532 条真实活动模板沉淀为结构化数据，通过倒排算法生成标准 ProjectData，再用 Web 画布和 ChatPanel AI 完成可视化、查询、编辑和诊断，把初版计划生成压缩到 1 小时以内。
>
> 这个作品不是一个只会聊天或展示的 AI demo。它的核心是“数据协议 + 确定性算法 + AI 交互”的组合：业务知识先被沉淀成可计算模板，计划结果以 JSON Schema 复用，AI 负责理解自然语言和辅助编辑，最终输出可以被画布、Skill、飞书和后续系统共同消费。

这版更稳：不自称满分，但把业务价值、技术差异化和评分命中点都交代清楚了。
