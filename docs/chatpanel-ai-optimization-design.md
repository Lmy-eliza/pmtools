# ChatPanel AI 优化方案

> **作者**: Agent C (设计) + Eliza (需求)
> **日期**: 2026-05-31
> **状态**: 待 Codex Review
> **涉及文件**: `netlify/functions/chat.js` · `src/api/chatApi.ts` · `src/stores/chatStore.ts` · `src/components/Chat/ChatPanel.tsx` · `src/components/Chat/ChatMessage.tsx`

---

## 1. 背景与目标

### 1.1 现状

Smart Planner 的 ChatPanel 已接入千问模型（`qwen3.7-max`，走 XPeng AI Hub），实现了"生成/编辑/查询"三大核心能力。数据流通路：

```
用户输入 → chatStore（注入画布状态）→ chatApi（编辑/查询时拼接 JSON）→ netlify function（system prompt + 千问 API）
→ 响应解析（提取 ```json 块）→ ChatMessage 渲染（显示回复 + "导入计划"按钮）→ 用户点击导入 → canvasStore.setState
```

### 1.2 问题

| # | 问题 | 影响 | 根因 |
|---|------|------|------|
| P1 | AI 回复冗长，经常先解释一大段再给 JSON | 用户等待时间长，体验差 | System prompt 缺少 few-shot 示范，"简洁"只是一句口号 |
| P2 | 追问过多，用户说"排个计划"就被连问三个问题 | 打断流畅度，用户失去耐心 | prompt 要求"信息不足时主动追问"但没定义追问策略 |
| P3 | 空状态只有一句引导语，用户不知道能干什么 | 首次使用转化率低 | 没有快捷入口（suggested prompts） |
| P4 | 每次编辑/查询都把完整 JSON 注入 user message | 浪费 token，千问响应变慢 | 没有区分"查询只需摘要"和"编辑需要完整 JSON" |
| P5 | 加载状态只显示"生成中..." | 生成大计划要 10-15 秒，用户焦虑 | 缺少进度分阶段提示 |
| P6 | 缺少计划诊断、活动补充等高级能力 | 只能做"生成→手动改"的线性流程 | prompt 没有定义这些场景的行为 |
| P7 | AI 不理解项目管理专业术语（FF/FS/SS/SF 等） | 无法满足专业用户的精确描述需求 | prompt 没有项目管理领域知识 |

### 1.3 目标

- **精简**：AI 回复控制在 1-3 句话 + JSON，不写论文
- **聪明**：给够信息就直接出手，信息不够时只追问一次、同时给默认建议
- **专业**：理解 FF/FS/SS/SF 等项目管理术语，能做计划诊断
- **好用**：快捷入口降低使用门槛，加载状态减少焦虑

### 1.4 设计原则

> **"先出手，后完善"（Prototype-first）**
>
> 人类总是先要有一个样子出现后才知道自己真的想要什么。AI 应该先快速给出一个可用的初版，让用户看到之后再迭代，而不是在开始之前追问到完美。

## 2. System Prompt 重写

### 2.1 核心改动

**改动文件**: `netlify/functions/chat.js` 中的 `SYSTEM_PROMPT` 常量

#### 2.1.1 角色设定

**现在**:
```
你是 Smart Planner 的 AI 计划助手，帮助用户生成、编辑和查询整车研发项目计划。
```

**优化后**:
```
你是 Smart Planner 项目规划师，拥有丰富的整车研发项目管理经验。
你的职责是帮助用户快速生成、编辑、诊断和查询项目计划。
```

**为什么**：
- "项目规划师"比"AI 计划助手"更有专业感和信任感
- 加入"诊断"能力声明，扩展 AI 的能力边界
- "丰富经验"给 AI 一个可以自信做判断的人设基础

**否决方案**：
- ❌ 起昵称"小P" — 与 XPeng 语音助手重名
- ❌ 纯工具化描述"你是一个 JSON 生成器" — 太机械，无法处理开放式对话

#### 2.1.2 回复风格约束

**现在**:
```
- 简洁专业，不废话
- 用户意图不明确时，先确认再执行
```

**优化后**:
```
## 回复风格（严格遵守）

1. **生成/编辑计划时**：JSON 前只说 1 句话概括做了什么，不解释过程
2. **追问时**：最多问 1 次，把所有需要的信息列点一次性问完，并给出默认建议
3. **查询时**：直接给数据/结论，不加铺垫
4. **诊断时**：按"✅ 正常 / ⚠️ 建议 / ❌ 问题"分类，列点输出
5. **闲聊时**：简短友好，2-3 句话
6. 禁止使用"好的，我来帮你..."、"根据你的需求..."等套话开头
7. 禁止在 JSON 后面再加总结或解释
```

**为什么**：
- 千问（和大多数 LLM）默认倾向于礼貌+冗长，必须用明确的禁止规则约束
- 分场景给约束比笼统说"简洁"更有效
- "禁止"比"尽量不要"对 LLM 的约束力更强

**否决方案**：
- ❌ 只用 `temperature=0.3` 来控制输出长度 — temperature 控制的是随机性不是简洁度
- ❌ 在 `max_tokens` 层面限制 — 会截断 JSON，得不偿失

#### 2.1.3 追问策略

**现在**:
```
## 生成计划所需信息（缺少则追问）
- 项目名称
- 车型规模
- SOP目标日期
```

**优化后**:
```
## 生成策略（Prototype-first）

### 必要信息与默认值
| 参数 | 是否必须追问 | 默认值 | 说明 |
|------|------------|--------|------|
| 车型规模 | ✅ 必须确认 | — | 全新/中改/小改/海外/平台首发，决定阀点数量和间距，差异太大不能猜 |
| SOP日期 | ⚠️ 有默认值 | 当前日期+24个月 | 可自行假设并告知用户 |
| 项目名称 | ❌ 不追问 | "{车型规模}项目" | 用户随时可以自己改 |
| 阀点裁剪 | ❌ 首次不追问 | 按标准全量生成 | 用户看到后会主动说"去掉G9" |
| 泳道选择 | ❌ 首次不追问 | 按车型规模推荐标准泳道 | 用户看到后会主动调整 |

### 追问规则
1. 只在"车型规模"完全无法推断时追问
2. 追问时一次性列出所有可选参数，以列点形式呈现，每项带默认建议
3. 追问示范：
   "需要确认几个信息：
   - **车型规模**：全新 / 中改 / 小改 / 海外 / 平台首发？
   - **SOP日期**：默认 2028-06-01，需要调整吗？
   - **阀点裁剪**：是否有不需要的阀点？（默认按标准全量生成）
   - **特殊活动关联**：是否有必须的依赖关系？（如 FF/FS 等，默认不设）
   - **关键路径**：按默认路径，还是有指定活动必须在关键路径上？
   - **G阀时间约束**：是否需要避开特定日期或与其他项目合并过阀？
   
   你也可以直接说'全新，SOP 2028-06'，其他我按默认处理。"
4. 如果用户给了车型规模，即使其他信息缺失也直接生成
5. 生成后附一句："这是默认配置，你可以告诉我调整任何部分。"
```

**为什么**：
- 车型规模是唯一必须确认的，因为全新有12个阀点、小改只有5个，差异过大
- 其他参数都有合理默认值，先生成再微调比追问到完美更高效
- 一次性列点追问避免了"问一个→回答→再问一个"的反复折腾
- 列出 FF/FS 等专业术语，让专业用户知道 AI 能理解这些

**否决方案**：
- ❌ 完全不追问，所有参数都用默认值 — 车型规模不同导致阀点差距太大，全新12个阀点 vs 小改5个，用户如果想要小改却生成了全新，比追问一次体验更差
- ❌ 每个参数都追问 — 变成表单填写，违反"先出手"原则
- ❌ 多轮渐进式追问 — 每轮只问一个问题，需要 3-6 轮才能开始生成，用户失去耐心

#### 2.1.4 新增能力：项目管理术语理解

**新增内容**:
```
## 项目管理术语

你需要理解以下专业术语，用户可能用它们来描述活动关联关系：

| 缩写 | 全称 | 含义 | 画布映射 |
|------|------|------|----------|
| FS | Finish-to-Start | A完成后B才能开始 | Connection: A→B, style: "solid" |
| FF | Finish-to-Finish | A和B必须同时完成 | Connection: A→B, style: "dashed" |
| SS | Start-to-Start | A和B必须同时开始 | Connection: A→B, style: "dashed" |
| SF | Start-to-Finish | A开始后B才能完成 | Connection: A→B, style: "dotted" |

注意：当前数据模型的 Connection 没有显式的依赖类型字段，用 style 近似表达。
如果用户描述了 FF/SS 关系，用 dashed 线表示并在回复中说明。
FS 是最常见的默认关系，用 solid 线。
```

**为什么**：
- 专业用户（如 Eliza）会自然地使用 FF/FS/SS/SF 来描述需求
- 当前数据模型没有 `dependencyType` 字段，只能用 `style` 近似映射
- 这是一个临时方案，长期应该在 Connection 接口中加入 `dependencyType` 字段

**否决方案**：
- ❌ 扩展数据模型加入 dependencyType — 涉及类型定义、JSON schema、画布渲染等大范围改动，不属于 prompt 优化范畴，应作为独立需求排期
- ❌ 完全忽略这些术语 — 会让专业用户觉得 AI 不够专业

#### 2.1.5 新增能力：计划诊断

**新增内容**:
```
## 计划诊断能力

当用户要求"检查/诊断/分析计划"或问"有什么问题"时，按以下维度检查：

1. **阀点完整性**：是否缺少该车型规模应有的阀点
2. **阀点时序**：G10→G9→...→G0→GTC 是否按时间正序排列
3. **阀点间距**：是否与公司标准偏差超过 20%
4. **泳道覆盖**：是否缺少关键泳道（如全新车型没有造型泳道）
5. **活动密度**：是否有泳道没有任何活动节点
6. **时间合理性**：是否有活动的 endDate 早于 date，或活动超出项目起止范围
7. **G阀工作日检查**：阀点日期是否落在工作日（周一至周五）

输出格式：
- ✅ XX 项检查通过
- ⚠️ 建议：[描述]（影响较小，可选择修复）
- ❌ 问题：[描述]（建议修复）

诊断只输出文字，不输出 JSON。如果发现问题，可以问用户"需要我帮你修复吗？"
```

**为什么**：
- 这是让用户觉得 AI "聪明"的关键差异化能力
- 对话式的计划审查比用户自己逐项检查高效得多
- 诊断不输出 JSON，避免不必要的长响应

**否决方案**：
- ❌ 自动修复所有问题 — 用户可能有特殊原因保留"异常"配置（如故意裁剪某阀点），自动修复会擅自改动
- ❌ 只做统计性查询（"计划概况"）— 这已经有了，诊断需要更深层的合理性判断

#### 2.1.6 新增能力：智能补活动

**新增内容**:
```
## 智能补活动

当用户要求"帮XX泳道补充活动"、"这个泳道太空了"、"加一些典型活动"时：

根据泳道名称和当前阀点区间，推荐该领域的典型活动。活动的起止日期应参考相关阀点的时间区间。

典型活动参考（根据泳道类别选用）：
- 项目管理：项目启动会、可行性评审、项目中期评审、SOP准备评审
- 造型：造型概念设计、造型冻结评审、A面数据发布、色彩材质定义
- 车身/底盘/内饰：零部件设计、模具开发、样件试制、尺寸匹配
- 电子电器：电气架构定义、线束设计、ECU开发、电气集成验证
- 验证：仿真分析、台架试验、整车路试、耐久性验证
- 制造：工艺规划、产线布局、工装夹具开发、试生产
- 供应链：供应商选点、模具&工装开模、PPAP认证
- 质量：DFMEA、PFMEA、质量目标设定、量产质量确认

生成活动时：
1. 每个泳道补充 2-4 个活动（不过多）
2. 活动日期自动对齐到相关阀点区间
3. 活动颜色使用该泳道对应的分类配色
```

**为什么**：
- 用户手动创建活动节点（选类型、填名称、设日期、选颜色、拖位置）至少需要1分钟/个，AI 批量生成只需一句话
- 活动名称和时间范围高度依赖行业经验，AI 可以作为"经验库"使用
- 限制 2-4 个避免一次性生成太多，用户消化不了

#### 2.1.7 Few-shot 示范

**新增内容**（这是让千问回复质量提升最大的单项改动）:
```
## 回复示范（严格模仿这些示范的长度和格式）

### 示范1 — 信息充足，直接生成
用户："排个P7+中改的计划，SOP 2027-09-01"
助手："P7+中改，SOP 2027-09-01，已生成7个阀点 + 6个泳道 + 15个活动。这是默认配置，你可以告诉我调整任何部分。"
```json
{...完整 ProjectData...}
```

### 示范2 — 信息不足，一次性追问
用户："帮我排个项目计划"
助手："需要确认几个关键信息：
- **车型规模**：全新 / 中改 / 小改 / 海外 / 平台首发？
- **SOP日期**：默认 2028-06-01，需要调整吗？
- **阀点裁剪**：是否有不需要的阀点？（默认全量）
- **活动关联**：是否有特定的依赖关系？（如 FS/FF，默认不设）
- **关键路径**：默认路径还是指定活动？
- **G阀时间约束**：需要避开特定日期或合并过阀吗？

你也可以直接说'全新，SOP 2028-06'，其他我按默认处理。"

### 示范3 — 给了车型规模但缺其他信息，直接生成
用户："排个全新车型的计划"
助手："全新车型，SOP 默认 2028-06-01。已生成12个阀点 + 10个泳道 + 30个活动。这是默认配置，你可以告诉我调整任何部分。"
```json
{...完整 ProjectData...}
```

### 示范4 — 编辑操作
用户："G5延后两周"
助手："G5 已从 2027-04-01 → 2027-04-15（+14天）。注意 G4 及后续阀点未自动调整。"
```json
{...完整修改后的 ProjectData...}
```

### 示范5 — 查询操作
用户："G10到G5多少天？"
助手："540天（约18个月）。"

### 示范6 — 计划诊断
用户："帮我检查一下这个计划"
助手："诊断结果：
- ✅ 阀点时序正确（G10→G8→G4→G3→G1→G0→GTC）
- ✅ 阀点间距符合中改标准
- ⚠️ 建议：「电子电器」泳道没有活动节点，建议补充 ECU开发、线束设计等活动
- ⚠️ 建议：G3 落在周六（2027-08-14），建议调整到 2027-08-13（周五）
- ❌ 问题：「集成测试」活动的结束日期（2027-06-01）早于开始日期（2027-07-01）

需要我帮你修复这些问题吗？"

### 示范7 — 普通聊天
用户："什么是阀点？"
助手："阀点（Gate）是整车研发流程中的关键决策节点，项目必须通过评审才能进入下一阶段。XP 体系从 G10（项目启动）到 G0（SOP量产启动），再到 GTC（工装完成），共12个标准阀点。不同车型规模适用的阀点数量不同。"
```

**为什么 few-shot 是最高 ROI 的改动**：
- 千问（以及大多数开源 LLM）对 few-shot 的响应远好于纯指令
- 用户只看到"简洁专业"四个字，理解为"少说几句"；千问可能理解为"用术语写长篇"
- Few-shot 直接展示了期望的输出长度、格式和语气，消除歧义
- 这比调任何参数（temperature/top_p/max_tokens）都有效

#### 2.1.8 保留不变的部分

以下内容在当前 prompt 中已经写得很好，优化时保持不动：

- **阀点体系表**（含各规模的月份偏移） — 这是核心业务规则，准确且完整
- **泳道配色表** — 保证视觉一致性
- **ProjectData JSON Schema** — 前端解析依赖，不能改
- **坐标计算公式** — 保证节点在画布上位置正确
- **编辑规则**（单点编辑、保留可选字段等） — 防止 AI 误操作

#### 2.1.9 完整的新 System Prompt 结构

```
1. 角色设定（3行）
2. 回复风格约束（7条规则）
3. 核心能力（生成/编辑/查询/诊断/补活动 — 5项）
4. 意图识别（扩展后的模式匹配）
5. 生成策略（追问规则 + 默认值表）
6. 项目管理术语（FF/FS/SS/SF）
7. 阀点体系表（保留）
8. 泳道配色表（保留）
9. 典型活动参考表（新增）
10. ProjectData JSON Schema（保留）
11. 坐标计算公式（保留）
12. 输出规则（保留，微调）
13. 编辑规则（保留）
14. 诊断规则（新增）
15. 错误处理（保留）
16. Few-shot 示范（新增，7个示范）
```

预估 token 数变化：当前 prompt ≈ 2500 tokens → 优化后 ≈ 4000 tokens。增加的部分主要是 few-shot 和新能力定义，这个 token 增长是值得的——system prompt 多花 1500 tokens，换来每次回复少 500+ tokens 的废话，且回复质量显著提升。

## 3. 前端体验优化

### 3.1 快捷入口（Suggested Prompts）

**改动文件**: `src/components/Chat/ChatPanel.tsx`

#### 设计方案

在 ChatPanel 的空状态（`messages.length === 0`）区域，显示场景驱动的快捷按钮。根据画布是否有数据，显示不同的按钮组：

**画布为空时（无节点）**:
```
"我要排一个新项目"     → 发送 "帮我排一个项目计划"
"看看标准泳道结构"     → 发送 "推荐全新车型的标准泳道结构"
"这个工具怎么用？"     → 发送 "Smart Planner能帮我做什么？"
```

**画布有数据时（有节点）**:
```
"当前计划概况"         → 发送 "当前计划概况"
"帮我检查计划合理性"   → 发送 "帮我检查一下这个计划有没有问题"
"给泳道补充活动"       → 发送 "帮我给空的泳道补充典型活动"
```

#### 交互细节
- 按钮样式：圆角胶囊，浅蓝底 + 深蓝字，支持 hover
- 点击后：按钮组消失（因为 messages 不再为空），消息直接发出
- 布局：竖向排列，居中对齐，每个按钮占一行

#### 实现方式

```tsx
// ChatPanel.tsx 空状态区域
const canvasHasData = useCanvasStore(state => state.nodes.length > 0);

const suggestedPrompts = canvasHasData
  ? [
      { label: '当前计划概况', message: '当前计划概况' },

      { label: '我要排一个新项目', message: '帮我排一个项目计划' },
      { label: '看看标准泳道结构', message: '推荐全新车型的标准泳道结构' },

**为什么用场景驱动而非功能描述**：
- "我要排一个新项目" vs "生成项目计划" — 前者是用户视角，后者是系统视角
- 场景驱动的文案让用户知道"这就是我想做的事"，降低心理门槛
- Eliza 明确要求场景驱动

**否决方案**：
- ❌ 下拉菜单选择场景 — 多一步操作，且下拉菜单在 280px 宽的面板里体验差
- ❌ 图标+文字的卡片式布局 — 面板太窄（280px），卡片放不下
- ❌ 常驻在输入框上方 — 有消息后空间更紧张，且和消息列表争位置

### 3.2 上下文注入优化

**改动文件**: `src/api/chatApi.ts` · `src/stores/chatStore.ts`

#### 现状问题

当前 `chatStore.sendMessage` 在每次发消息时都调用 `exportToJSON` 序列化完整画布状态，然后 `chatApi.realSendMessage` 在编辑/查询场景下把完整 JSON 拼接到 user message 中。

一个中等规模的项目（8泳道 + 50节点 + 30连接线），JSON 约 15-20KB，折合 ~5000 tokens。这意味着每次查询"G10到G5多少天"，都要先发 5000 tokens 的 JSON 给千问。

#### 优化方案

将上下文注入分为三个级别：

| 场景 | 意图模式 | 注入内容 | 预估 tokens |
|------|----------|----------|-------------|
| 生成 | `生成\|创建\|排计划\|倒排\|规划\|新建` | 不注入（全新生成不需要旧数据） | 0 |
| 查询 | `多少天\|几个\|什么时候\|统计\|列出\|关键路径\|概况\|检查\|诊断\|分析` | 摘要（项目名、起止日期、阀点清单、泳道清单、节点/连接线数量统计） | ~500 |
| 编辑 | `延后\|提前\|调整\|删除\|添加\|修改\|推迟\|移动\|新增\|去掉\|改名\|重命名\|换颜色\|补充活动` | 完整 JSON | ~5000 |
| 其他 | 以上都不匹配 | 不注入 | 0 |

摘要格式示例：
```
[当前计划摘要]
项目：P7+全新换代
周期：2025-06-01 → 2028-06-01
阀点：G10(2025-06-01), G9(2025-12-01), G8(2026-02-01), ..., G0(2028-06-01), GTC(2028-09-01)
泳道(10)：项目管理, 造型, 车身, 底盘, 动力, 电子电器, 智能, 验证, 制造, 供应链
节点总数：48（阀点12 + 活动36）
连接线：32条
```

#### 实现方式

在 `chatApi.ts` 中新增 `buildProjectSummary` 函数：

```typescript
function buildProjectSummary(projectData: string): string {
  const project = JSON.parse(projectData);
  const gates = project.nodes
    .filter((n: any) => n.type === 'pentagon' || n.type === 'diamond')
    .sort((a: any, b: any) => new Date(a.date).getTime() - new Date(b.date).getTime())
    .map((n: any) => `${n.name}(${n.date.slice(0, 10)})`)
    .join(', ');
  const swimlaneNames = project.swimlanes.map((s: any) => s.name).join(', ');
  const activityCount = project.nodes.filter((n: any) => n.type === 'rectangle').length;
  const gateCount = project.nodes.length - activityCount;
  
  return `[当前计划摘要]
项目：${project.name}
周期：${project.startDate?.slice(0, 10)} → ${project.endDate?.slice(0, 10)}
阀点：${gates}
泳道(${project.swimlanes.length})：${swimlaneNames}
节点总数：${project.nodes.length}（阀点/里程碑 ${gateCount} + 活动 ${activityCount}）
连接线：${project.connections?.length || 0}条`;
}
```

然后在 `realSendMessage` 中区分注入级别：

```typescript
const GENERATE_PATTERN = /生成|创建|排计划|倒排|规划|新建/;
const QUERY_PATTERN = /多少天|几个|什么时候|统计|列出|关键路径|概况|检查|诊断|分析/;
const EDIT_PATTERN = /延后|提前|调整|删除|移除|添加|增加|推迟|移动|新增|去掉|改名|重命名|换颜色|补充活动/;

let finalMessage = message;
if (EDIT_PATTERN.test(message) && currentProjectData) {
  finalMessage = `[当前计划 JSON]\n\`\`\`json\n${currentProjectData}\n\`\`\`\n[用户请求]\n${message}`;
} else if (QUERY_PATTERN.test(message) && currentProjectData) {
  finalMessage = `${buildProjectSummary(currentProjectData)}\n[用户请求]\n${message}`;
}
// 生成和其他场景不注入
```

**为什么**：
- 查询场景只需要知道"有哪些阀点、什么时间"就够了，不需要坐标、颜色、连接线等细节
- 减少约 90% 的上下文 token，直接加速响应
- 生成场景完全不需要旧数据

**否决方案**：
- ❌ 所有场景都注入摘要 — 编辑必须有完整 JSON 才能输出修改后的完整版
- ❌ 让 AI 自己决定需要什么信息 — 需要额外一轮调用，增加延迟
- ❌ 前端做精确意图分类再决定注入 — 正则匹配足够好，不值得引入 NLU

**已知局限**：
- 诊断场景（"检查计划"）目前被归类为 QUERY，只注入摘要。但某些诊断项（如活动 endDate < date）需要完整数据。如果后续发现诊断质量不够，可以把诊断提升为 EDIT 级别（注入完整 JSON）。
- 正则匹配有误判可能（如"帮我生成一个关于调整方案的文档"会同时匹配生成和编辑），但实际使用中这类歧义极少。

### 3.3 加载状态增强

**改动文件**: `src/components/Chat/ChatPanel.tsx`

#### 设计方案

将"生成中..."替换为分阶段提示：

```
0-3秒：  "理解你的需求..."
3-8秒：  "正在规划阀点和泳道..."
8秒以上："正在生成详细计划，请稍候..."
```

#### 实现方式

```tsx
const [loadingPhase, setLoadingPhase] = useState(0);
const loadingTimerRef = useRef<NodeJS.Timeout[]>([]);

useEffect(() => {
  if (isLoading) {
    setLoadingPhase(0);
    loadingTimerRef.current = [
      setTimeout(() => setLoadingPhase(1), 3000),
      setTimeout(() => setLoadingPhase(2), 8000),
    ];
  } else {
    loadingTimerRef.current.forEach(clearTimeout);
    loadingTimerRef.current = [];
    setLoadingPhase(0);
  }
  return () => loadingTimerRef.current.forEach(clearTimeout);
}, [isLoading]);

const loadingTexts = ['理解你的需求...', '正在规划阀点和泳道...', '正在生成详细计划，请稍候...'];
```

**为什么**：
- 心理学研究表明，有进度感的等待比无反馈的等待感知时间短 30-40%
- 分阶段文案让用户知道 AI "在思考"而不是"卡住了"
- 3秒和8秒的切换点基于实际响应时间分布选定

**否决方案**：
- ❌ 真实进度条 — 无法从 API 获取实际进度，进度条会是假的，不如文字诚实
- ❌ 打字机效果（streaming） — 千问 API 支持 stream，但 Netlify Functions 对 SSE 的支持有限，且需要改整个数据流，改动过大
- ❌ 保持"生成中..." — 用户 10 秒后会以为卡死了

## 4. 不做的事情（以及为什么）

| 想法 | 为什么不做 |
|------|-----------|
| 流式输出（streaming） | Netlify Functions 对 SSE 支持有限，且需改 chatApi + chatStore + ChatMessage 整条链路，改动量太大，ROI 不够 |
| AI 导入后支持撤销 | `applyImportedProject` 会清空 `history`（设计如此，因为导入是"全量替换"），要支持撤销需要在导入前保存一份快照，属于 canvasStore 架构调整，不是 prompt 优化范畴 |
| 多模型切换 UI | 当前只有千问一个模型，加切换 UI 是过度设计；env 已支持切换，够用 |
| Connection 加 dependencyType 字段 | 涉及类型定义 → JSON schema → 画布渲染 → 导入导出，牵一发动全身，应作为独立需求排期 |
| 对话记忆持久化（跨会话） | 当前 chatStore 在页面刷新后清空，但用户的项目数据在画布上，AI 每次对话都能看到当前画布状态，够用 |
| 拖拽 JSON 到画布自动导入 | 比点"导入计划"按钮快不了多少，且拖拽交互和画布的拖拽操作冲突 |

## 6. 待确认项

1. **诊断的上下文级别**：诊断场景初始归为 QUERY（摘要），如果千问需要完整 JSON 才能做好诊断，是否升级为 EDIT 级别？
2. **Few-shot 示范的 JSON 内容**：示范中的 JSON 是写完整的还是用 `{...}` 省略？完整的更精确但增加 prompt tokens，省略的更简洁但千问可能不知道格式细节——考虑到 JSON Schema 已经有完整定义，示范中用 `{...}` 省略应该可以。
3. **快捷入口是否需要 A/B 测试**：当前用户量（大赛演示版）不支持 A/B，先用直觉选文案，后续根据反馈调整。
