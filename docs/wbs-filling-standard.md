# WBS 计划填写规范 v1.0

> 适用范围：Smart Planner 活动模板库的数据录入与维护
> 目标：统一不同编写者的填写习惯，确保 AI 系统可准确解析、生成和编辑项目计划

---

## 1. 活动命名规范

### 格式要求

**标准格式**：`[动词] + [对象] + [状态/成果]`

| 示例 | 评价 |
|------|------|
| ✅ 模具验证完成 | 动词+对象+状态，语义明确 |
| ✅ B版材料清单发布 | 版本+对象+动作，可追溯 |
| ✅ PT1整车试验报告评审 | 阶段+对象+动作，层级清晰 |
| ❌ 模具 | 纯名词，无法判断活动内容和完成标志 |
| ❌ 处理相关工作 | 模糊动词+笼统对象，无实际信息量 |
| ❌ D11ES-G5阀点ET数据冻结 | 活动名称不应包含项目代号，模板需通用 |

### 命名禁忌

- 不含项目代号（如 D11ES、P7i）——模板应跨项目通用
- 不超过 25 个汉字——过长影响画布渲染和可读性
- 不使用模糊动词（处理、跟进、推进、配合）——改用具体动作（评审、发布、交付、完成、冻结）
- 不在名称中嵌入日期或时间信息——时间由结束周和活动周期字段表达

## 2. 层级划分标准

| 层级 | 定义 | 工期范围 | 典型特征 | 示例 |
|------|------|---------|---------|------|
| 0 | 阀点级 | — | 里程碑事件，标记阶段转换 | G5 ET启动、G0 SOP |
| 1 | 一级活动（管理级） | 4-16 周 | 可独立交付、可分配责任部门、跨专业协调 | ET订单审批、整车DVP试验 |
| 2 | 二级活动（执行级） | 1-8 周 | 具体任务、单一责任人、明确交付物 | 线束TG2数据发布、底盘标定完成 |

### 划分原则

- **一条记录 = 一个活动实例**，不合并跨阀点的同名活动
- 如果一个活动跨越两个阀点区间，必须拆分为两条记录，分别挂到对应阀点
- 层级 0 不需要手动填写——阀点由阀点定义表管理
- 层级 1 和层级 2 的区分依据：是否需要拆解才能分配给单一责任人

## 3. 时间填写规范

### 结束周（相对所属阀点）

- **含义**：该活动在所属阀点之前/之后多少周完成
- **负数**：活动在阀点之前完成（绝大多数情况）
- **0**：活动在阀点当周完成
- **正数**：活动在阀点之后完成（极少见，需在备注中说明原因）

| 示例 | 含义 |
|------|------|
| -10 | 在所属阀点前 10 周完成 |
| 0 | 在阀点当周完成 |
| -2 | 在阀点前 2 周完成 |

### 活动周期（周）

- **含义**：活动持续时长，单位为周
- **必须 > 0**——不存在工期为 0 的非里程碑活动
- **精度**：0.5 周（即可填 1.5、2.5 等半周值）
- Agent 通过 `开始日期 = 结束日期 - 活动周期 × 7天` 自动推算开始日期

### 禁忌

- ❌ 模糊表述（"大约3周"、"2-4周"）——一律填精确数字
- ❌ 活动周期为 0——如果工期为零，应标记为里程碑（节点类型=五边形）
- ❌ 结束周为正数但未说明原因——正数意味着活动在阀点之后才完成，需备注理由

## 4. 甬道分配规范

### 基本规则

- 每个活动**必须且只能属于一个甬道**
- 甬道代表**组织职能**（部门/专业领域），不是项目阶段
- 甬道名称必须从甬道定义表的 33 个标准甬道中选择，不可自创

### 甬道与部门的映射

| 分类 | 甬道 |
|------|------|
| 管理 | 项目管理、产品、市场、财经、法务 |
| 工程 | 技术管理、属性集成、功能集成、整车架构、物理集成、仿真分析、热管理、动力 |
| 造型 | 外造型、内造型 |
| 零部件 | 车身外饰、内饰、材料、底盘 |
| 电子 | 电子电器、嵌入式 |
| 智能 | 通用智能、互联网、数据智能 |
| 验证 | 测试 |
| 供应链 | 采购 |
| 质量 | SQE、质量 |
| 制造 | 制造、KD中心 |
| 法规 | 法规认证、法规清单 |
| 服务 | 售后 |

### 常见错误

- ❌ 同一活动分配到多个甬道——拆分为多条记录
- ❌ 使用非标准甬道名称（如"研发"、"设计"）——映射到标准名称
- ❌ 甬道为空——每条记录必须有所属甬道

## 5. 依赖关系填写规范

### 依赖类型

| 类型 | 含义 | 使用频率 | 说明 |
|------|------|---------|------|
| **FS**（Finish-to-Start） | 前置完成后，后续才能开始 | **≥95%** | 默认类型，最常用 |
| SS（Start-to-Start） | 前置开始后，后续才能开始 | 少 | 两个活动可同时启动 |
| FF（Finish-to-Finish） | 前置完成后，后续才能完成 | 少 | 两个活动需同时结束 |
| SF（Start-to-Finish） | 前置开始后，后续才能完成 | 极少 | 特殊场景，使用时必须备注原因 |

### Lag 填写

- **正数**：间隔天数（前置完成后等 N 天再开始后续）
- **负数**：提前天数（前置完成前 N 天就开始后续，即重叠）
- **0**：紧接（前置完成后立即开始后续）

### 规范要求

- 使用 SS / FF / SF 时，**必须在备注中说明原因**
- 依赖关系必须在依赖关系表中建立，不要在备注中用文字描述
- 避免循环依赖（A→B→C→A）
- 一个活动可以有多个前置活动，但应避免不必要的冗余依赖

## 6. 其他字段填写要求

### 适用项目规模

- 多选字段，从 `平台首发 / 全新 / 中改 / 小改 / 海外` 中选择
- 至少选一个——如果一个活动不适用于任何规模，应设为"不启用"
- "换代" = "全新"，不要使用"换代"

### 节点类型

- 大部分活动为 `矩条形（普通活动）`
- 活动名称含「评审」「审批」「决策」「签发」「确认」→ `菱形（决策/评审节点）`
- 活动名称含「完成」「启动」「发布」「交付」「SOP」→ `五边形（里程碑）`
- 优先级：里程碑 > 决策/评审 > 普通活动

### 是否关键路径

- 仅标记在标准模板中**通常**位于关键路径上的活动
- 实际项目的关键路径由 Agent 基于依赖关系自动计算（CPM），此标记仅供参考

## 7. 常见错误清单

| 错误类型 | 描述 | 正确做法 |
|---------|------|---------|
| ❌ 跨阀点活动 | 同一活动横跨两个阀点 | 拆分为两条记录 |
| ❌ 零工期非里程碑 | 工期为 0 但不是里程碑 | 填写实际工期或改为里程碑 |
| ❌ 缺少甬道 | 所属甬道为空 | 必须分配到一个标准甬道 |
| ❌ 名称含项目代号 | "D11ES-ET数据冻结" | 改为"ET数据冻结"（模板通用） |
| ❌ 模糊工期 | "2-4周" | 填精确值如"3" |
| ❌ 多甬道归属 | 一个活动标记了多个甬道 | 拆分为多条记录 |
| ❌ 重复依赖 | A→B 和 A→C→B 同时存在 | 删除冗余的 A→B |
| ❌ 计划开始晚于结束 | 开始日期 > 结束日期 | 检查结束周和活动周期 |
| ❌ 使用非标枚举值 | 甬道名/阀点名不在标准列表 | 映射到标准值 |

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 活动名称 | 文本 | ✅ | 动词+对象+状态，≤25字 |

| 适用项目规模 | 多选 | ✅ | 至少选一个 |
| 活动层级 | 单选 | ✅ | 1=管理级，2=执行级 |
| 节点类型 | 单选 | ✅ | 按命名关键词自动分类 |
| 是否关键路径 | 单选 | 建议 | 标准模板中的参考标记 |
| 责任部门 | 文本 | 建议 | 负责执行的部门 |
| 交付物 | 文本 | 建议 | 活动完成的产出物 |
| 是否启用 | 单选 | 自动 | 默认"启用" |
