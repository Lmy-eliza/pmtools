## 🔴 P0 — 必须对齐后才能建表

### Q1：依赖关系是一对多，当前字段结构撑不住

**现状**：活动总表用 `前置活动`（关联多个）+ `依赖类型`（单选）来描述依赖。

**问题**：一个活动有多个前置时，每条依赖的类型和 Lag 可能不同。例如活动 C 依赖 A 是 `FS+7`、依赖 B 是 `FF+0`，但 `依赖类型` 只有一个单选格，无法区分哪条依赖是什么类型。

**建议方案**：新建 **「依赖关系表」**，每行一条依赖边：

| 字段 | 类型 | 示例 | 说明 |
|------|------|------|------|
| 后续活动 | 关联→活动总表 | ACT-008 | 被依赖方 |
| 前置活动 | 关联→活动总表 | ACT-003 | 前置方 |
| 依赖类型 | 单选 | FS / SS / FF / SF | 四种标准类型 |
| Lag（天） | 数字 | 7 | 正=延迟，负=提前（Lead），0=紧接 |

**请确认**：是否接受这个拆表方案？如不接受，请提供替代方案说明如何处理一对多依赖 + 每条依赖独立 Lag 的场景。

### Q2：Lag/Lead 天数字段缺失

**现状**：需求文档没有 Lag 字段。

**问题**：Eliza 原始需求明确提到 `FS+7`、`FF+7` 这种带偏移量的依赖写法。没有 Lag 字段，Agent 只能生成 `FS+0` 的无延迟依赖，不符合实际项目排期。

**请确认**：是否补充 Lag 字段？如果放在 Q1 的依赖关系表中则自然解决；如果不拆表，需要说明 Lag 信息存在哪里。

---

### Q3：时间基准矛盾——字段写"距G0月数"，流程写"从SOP倒推"

**现状**：
- 阀点定义表字段名：`平台首发/全新-距G0月数`（以 G0 项目关闭为锚点）
- Agent 流程描述：「从SOP倒推各阀点绝对日期」（以 SOP 即 G1 为锚点）

**问题**：
- G1 = SOP，G0 = 项目关闭（时间在 SOP **之后**）
- 如果字段是"距G0月数"，Agent 拿到用户输入的 SOP 日期后，需要先算出 G0 日期，再用 G0 日期加上各阀点的偏移——多一层间接转换，容易搞混正负方向
- 而且用户输入的是 SOP 不是 G0，以 G0 为锚点不直观

**建议**：字段改名为 `距SOP(G1)月数`，值也相应调整。Agent 直接 `SOP日期 + 偏移月数` 计算，无需中转。

**请确认**：时间锚点用 G0 还是 G1(SOP)？如果坚持用 G0，请说明 Agent 如何从 SOP 推算 G0 日期的具体规则。

## 🟡 P1 — 强烈建议对齐

### Q4：工期单位——天 vs 周

**现状**：
- Eliza 原始需求：「基准工期单位是**天**」
- 需求文档字段：`开始周（相对阀点）`、`结束周（相对阀点）`，默认工期单位为"周"

**问题**：单位不一致，建表时不知道按哪个来。

**建议**：统一用 **天** 作为基础粒度。字段改为 `开始天（相对阀点）`、`结束天（相对阀点）`、`基准工期（天）`。周 = 天÷7，可以用公式字段派生。天粒度更精确，Agent 计算也更简单。

**请确认**：同意改为天，还是坚持用周？如果用周，Eliza 要求的"基准工期单位是天"怎么满足？

## 🟢 P2 — 建议优化

### Q6：活动ID 格式——自动编号 or 手动文本？

**现状**：字段类型写"自动编号/文本"，示例写 `ACT-001`。

**问题**：
- 飞书自动编号的格式不一定能自定义成 `ACT-001`
- 依赖关系表要引用这个 ID，如果是自动编号，关联字段能不能正常工作？

**建议**：用 **文本字段** 手动编号，格式统一为 `ACT-XXX`。Agent 生成时按此格式识别。

---

### Q7："是否关键路径"的"按项目判断"选项，Agent 怎么处理？

**现状**：选项为 `是 / 否 / 按项目判断`。

**问题**：Agent 遇到"按项目判断"时的行为是什么？
- 默认当"否"输出？
- 跑 CPM 算法自行判断？
- 标记为 `unknown` 让用户手动确认？

**请给出明确规则**，否则 Agent 会不知道怎么处理这个值。

---

### Q8：连接线样式建议移到渲染层

**现状**：活动总表有 `连接线样式`（solid / dashed / dotted）。

**建议**：这是纯视觉属性，和项目管理逻辑无关。建议从数据模板中移除，改在画布渲染逻辑里按依赖类型自动映射：
- FS → solid
- SS / FF → dashed
- SF → dotted

这样数据层更干净，维护成本更低。

**请确认**：同意移除，还是有其他需要保留的理由？
