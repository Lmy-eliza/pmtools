# 工作方法论
- **第一性原理**：聚焦最核心、最基础的产品设计逻辑，先搞清楚根本需求，再展开设计；拓展功能参考市场现有最好的甚至跨行业的产品并加以创新；
- **金字塔原理**：先给结论，再给依据。比如有3个方案A/B/C，先推荐方案A并说明理由，再对比其他方案的优劣势；
- **先规划后执行**：不要想到一步就马上做一步，以免返工浪费token。成体系地沟通，逐项沟通完成后再统一按规划执行，不把简单的事情复杂化；
- **信息安全意识**：一旦涉及公网发布、信息读取等存在信息泄露风险的事情，处理会非常谨慎并主动和用户商量，选择最安全的方式；
- **参考最佳实践**：对于用户的输入加以处理，形成条理清晰的输入。用户的知识面不一定全面，不假设用户清楚全部需求，习惯和用户共同讨论，聊清楚根本需求和细节后再执行；
- **环境感知约束** ：本项目运行在 Windows 环境。执行操作前需确认是否存在 `.git` 文件夹，若无则提示我执行 `git init` 激活仓库。

## Git 版本管理军规
- **提交格式 (Conventional Commits)** ：遵循 `<类型>(<作用域>): <描述>` 格式（例如 `feat(ui): 增加日期自动更新功能`），其中：`feat`: 新功能 | `fix`: 修复 | `docs`: 文档 | `refactor`: 重构 | `chore`: 辅助工具变动；
- **语言要求** ：Git 提交信息（Commit Message）必须使用 **中文**；
- **安全发布** ：严禁在未授权情况下执行 `git push --force`；执行 `git push` 前必须确认远程分支名称；
- **冲突处理** ：遇到冲突先进入 `Plan` 模式列出冲突点，由我确认后再合并;
- **文件过滤规则** ：由于 Windows 系统限制，已通过 .gitignore 屏蔽 .claude/ 缓存文件夹，防止 nul 文件导致索引失败；
- **身份标识** ：本地已全局配置 user.name 和 user.email，确保提交记录合法。

## 自动化操作 (Hooks)
- **初始化检查 **：启动时若检测到无 .git 文件夹，需主动执行 /init 或提示我运行 git init；
- **自动化军规 (Pre-Commit) **：在执行 git commit 前，你必须自动运行 npm run lint (或对应的自检脚本)，若报错，必须先修复报错再尝试提交；
- **质量复核 (Slash Command) **：完成复杂功能后，我会执行 /review，你需对修改后的代码进行二次审计，寻找潜在 Bug；
- **Pre-Tool-Use (调用工具前)** ：在执行关键修改前，需检查相关文件的引用关系；
- **Pre-Commit (提交前)** ：在执行 `git commit` 前，必须自动运行 `lint` 或测试脚本，确保代码质量；
- **Post-Compact (压缩后)** ：执行 `/compact` 命令后，需简要概括当前项目进度，保持逻辑连续；
- **Git 仓库路径**：https://gitlab.x-peng.com/millionwinwin/personal-work-timer.git
- **鉴权模式**：强制使用 HTTPS + PAT (令牌)，已避开不可用的 SSH 通道

## 版本管理规范
- **版本号**：自然数递增，v1, v2, v3...（不使用 v1.0 格式）；
- **命名规则**：`plan-{日期}-{描述}-v{N}.md`（例：`plan-20260318-热力图增强-v1.md`）；
- **保留策略**：每次生成新计划都保留，不覆盖旧版本，新计划使用新版本号；
- **锁板规则**：产品开发完成后，基于过往计划形成最终完整计划，锁板命名为 `plan-{日期}-completed.md`，锁板后不再修改；
- **迭代保留**：所有迭代版本保留，直到确认最终验收。
